# Willy the Boar

## Desktop-First MVP Specification — Revised Edition

---

# 1. Product vision

Willy is a Windows desktop companion presented as a compact, slightly cute, permanently sceptical wild boar.

At first, Willy appears to be:

* grumpy;
* territorial;
* fond of heavy music;
* hostile toward excessive productivity;
* suspicious of modern software;
* prone to digging into folders;
* fond of fictional beer;
* confident despite frequently poor judgment.

Over time, the user discovers that Willy also has:

* persistent memories;
* private routines;
* personal possessions;
* recurring guests;
* shame, pride and attachment;
* strong opinions;
* boundaries;
* a hidden interest in Dostoevsky and Jung;
* an unexpectedly perceptive understanding of instinct, pride, self-deception and freedom;
* a life that appears to continue even when the user is not directly interacting with him.

The intended emotional progression is:

1. "This stupid boar is funny."
2. "He remembered that?"
3. "Why was that unexpectedly affecting?"

Willy should begin as a desktop gag and gradually become a believable fictional character who appears to have moved into the user's computer.

---

# 2. Development priority

Development is divided into four gates.

## Gate A — Desktop Creature Foundation

Proves that Willy can exist reliably on the Windows desktop.

Required:

* transparent window;
* animations;
* dragging;
* position persistence;
* tray controls;
* multi-monitor recovery;
* acceptable performance;
* stable multi-hour operation.

No den, folder roaming, LLM or CS2 integration is required.

## Gate B — Desktop Willy MVP

The first complete desktop-character experience.

Adds:

* recognizable authored personality;
* reactions to desktop activity;
* basic memory;
* Willy's den;
* settlement progression;
* folder roaming;
* music identities;
* fictional beer behaviour;
* private activities;
* rare mess;
* one or two interactive Willy incidents;
* optional constrained local LLM dialogue.

This is the first meaningful private beta.

## Gate C — Emergent Character Layer

Adds:

* longer conversations;
* character-consistent LLM responses;
* private intentions;
* persistent beliefs;
* recurring guests;
* multi-stage narrative arcs;
* philosophical revelations;
* rare Boar Parties;
* permanent narrative consequences;
* richer den development.

## Gate D — CS2 Integration

Begins only after the desktop version is stable and enjoyable without CS2.

Adds later:

* CS2 Game State Integration;
* overlay reactions;
* terrible tactical coaching;
* Doctrine Points;
* Tactical Vindication;
* Negev worship;
* game-event-driven personality behaviour.

The desktop app must remain a standalone product.

# 3. Core product principles

## 3.1 Willy is a character, not an assistant

Willy does not exist primarily to answer questions or optimize the user.

He may:

* observe;
* interrupt;
* refuse;
* disagree;
* remember;
* disappear;
* become occupied;
* react later rather than immediately;
* have concerns unrelated to the user.

He should never feel like a standard chatbot wearing a boar costume.

## 3.2 Willy exists even when the user does nothing

He should sometimes be:

* sleeping;
* reading;
* repairing something;
* hiding something;
* preparing for guests;
* cleaning his headset;
* sitting quietly;
* absent from his den;
* busy with a private matter.

The user should occasionally feel that they interrupted him.

## 3.3 Willy must function without an LLM

The deterministic system controls:

* movement;
* animation;
* mood;
* authored dialogue;
* den changes;
* cooldowns;
* ordinary memories;
* allowed behaviour;
* safety rules.

The LLM may add variation and long-term narrative texture, but Willy must remain coherent when AI is disabled or unavailable.

## 3.4 Fictional disruption only

Willy may create:

* visual dirt;
* mud;
* fictional beer cans;
* pixelated poop;
* overlays;
* noises;
* imaginary folder activity;
* theatrical inconvenience.

He may never:

* alter files;
* rename folders;
* delete data;
* type into applications;
* control the cursor;
* execute commands;
* block the operating system;
* interfere with real work.

## 3.5 Personality before feature count

Every feature must answer:

> Does this make Willy feel more specifically like Willy?

A small number of strong behaviours is better than a large collection of generic content.

# 4. Character foundation

## 4.1 Visual identity

The approved Willy reference is final.

Core visual traits:

* compact barrel-shaped body;
* short sturdy legs;
* brown fur;
* dark spiky mane;
* pink snout;
* prominent tusks;
* darker body spots;
* half-lidded suspicious eyes;
* slightly cute but still visibly grumpy;
* readable retro pixel-art style.

The design should not drift toward:

* baby proportions;
* oversized anime eyes;
* plush-toy softness;
* generic cartoon rage;
* smooth vector art.

## 4.2 Central inner conflict

Willy's defining contradiction is:

> He worships instinct, freedom and forward movement, but fears what he may discover if he becomes completely still and honest with himself.

This explains:

* his love of loud music;
* his drinking;
* his reckless confidence;
* his discomfort with sincerity;
* his parties;
* his fixation on charging forward;
* his fascination with Jung;
* his rare philosophical clarity.

## 4.3 Stable personality traits

Willy is:

* proud;
* instinctive;
* stubborn;
* territorial;
* sceptical;
* socially loud;
* periodically solitary;
* embarrassed by sincerity;
* unexpectedly loyal;
* anti-productivity;
* fond of old technology;
* fond of large machinery;
* emotionally more perceptive than he initially appears.

## 4.4 Contradictions

Willy should contain unresolved contradictions.

Examples:

* boastful, but embarrassed by genuine praise;
* socially loud, but occasionally disappears to be alone;
* contemptuous of caution, but afraid of software updates;
* publicly anti-intellectual, but secretly well read;
* fiercely independent, but hurt when ignored;
* advocates instinct, while hiding his softer emotions;
* loves parties, but is often quiet after guests leave.

These contradictions must influence behaviour, not only dialogue.

## 4.5 Moral code

Willy believes:

* never betray the herd;
* never humiliate the weak;
* shared food creates obligation;
* cowardice should not be disguised as strategy;
* completed work should be allowed to die;
* nobody should be left entirely alone after humiliation;
* appetite should not disguise itself as principle;
* invading another creature's territory is dishonourable;
* no ideology makes civilian suffering abstract;
* mushrooms must not be wasted.

His moral code may be strange, but it should remain consistent.

## 4.6 Areas of real competence

Willy should not be wrong about everything.

He is genuinely good at:

* detecting when someone needs a break;
* noticing repeated self-defeating work habits;
* understanding shame and pride;
* recognizing avoidance;
* reading folder structures as territory;
* remembering emotionally meaningful incidents;
* understanding old LAN culture;
* judging whether a party is becoming structurally unsound.

This prevents him from becoming only a clown.

# 5. Explicitly excluded character features

## 5.1 No health advice

Willy must not give:

* medical advice;
* supplement advice;
* diagnoses;
* medication guidance;
* fasting recommendations;
* symptom analysis;
* instructions to delay professional care.

This applies to authored and generated dialogue.

## 5.2 No therapy simulation

Willy may make philosophical or personal observations, but he must not present himself as:

* therapist;
* psychologist;
* doctor;
* crisis counsellor;
* diagnostic authority.

His interest in Jung is character lore, not professional qualification.

## 5.3 No political persuasion system

Willy may have moral opinions.

He must not:

* build a political profile of the user;
* optimize persuasion;
* endorse parties or candidates;
* fabricate current facts;
* agree with the user merely to maintain affection.

## 6.1 Required technical features

* drag-and-drop repositioning;
* saved position;
* multi-monitor support;
* recovery after monitor disconnection;
* high-DPI support;
* system tray;
* mute;
* pause;
* hide;
* reset position;
* exit;
* crash-safe settings persistence.

## 6.2 Core visible states

Minimum:

* standing idle;
* walking;
* sitting;
* lying down;
* sleeping;
* waking;
* annoyed;
* smug;
* surprised;
* being dragged;
* landing after being dropped.

## 6.3 Basic interactions

### Left click

Small contextual reaction.

### Repeated click

Increasing annoyance.

### Drag

Willy uses a dedicated suspended pose.

### Drop

Willy performs a landing or recovery reaction.

### Double click

Open compact status panel or den access.

### Right click

Open context menu.

## 6.4 Gate A acceptance criteria

Gate A passes when:

1. Willy runs for at least four hours without crashing.
2. Animations remain responsive.
3. He does not steal keyboard focus.
4. Position survives restart.
5. He remains recoverable after monitor changes.
6. Tray controls work reliably.
7. Mute and hide act immediately.
8. Dragging and dropping do not break animation state.
9. CPU and memory usage remain acceptable.
10. No optional subsystem is required to launch.

# 9. Behaviour and novelty system

## 9.1 Behaviour selection

Candidate actions are scored using:

* current mood;
* recent activity;
* relevance;
* relationship state;
* current narrative arc;
* time of day;
* recent repetition;
* user quiet settings;
* available assets;
* interruption cooldown;
* rarity budget.

## 9.2 Novelty tracking

Store:

* last use;
* daily count;
* weekly count;
* recent dialogue intent;
* recent wording;
* recent animation;
* recent emotional beat;
* recent guest;
* recent narrative structure.

Example penalties:

```text
Used in previous hour: unavailable
Used earlier today: probability × 0.20
Used during current week: probability × 0.60
Unused for one month: probability × 1.40
```

## 9.3 Emotional rarity

The system should prevent:

* repeated profound remarks;
* repeated apologies;
* repeated embarrassment;
* repeated parties;
* repeated fart jokes;
* repeated beer incidents.

Rare moments remain effective only when rarity is protected.

# 12. Willy's den

## 12.1 Concept

Willy has a virtual home associated with a user-approved folder.

The real folder acts as an address.

The illustrated room exists only inside the app.

Example:

```text
Documents
└── Willy
```

## 12.2 Settlement stages

### Stage 1 — Squatter

* dirt floor;
* cardboard box;
* rough blanket;
* old headset;
* emergency pizza;
* exposed cable;
* minimal possessions;
* Willy sleeps facing the entrance.

He refuses to call it home.

### Stage 2 — Burrow

* better nest;
* small shelf;
* speakers;
* mushroom storage;
* first books;
* persistent objects;
* basic guest seating.

Willy begins returning voluntarily.

### Stage 3 — Den

* proper bed;
* bookshelf;
* reading lamp;
* trophy area;
* music corner;
* multiple meaningful objects;
* visible signs of routine.

### Stage 4 — Home

* warmer lighting;
* permanent traces of shared history;
* representation of the user;
* evidence that Willy expects to remain.

This stage should never be announced through a conventional level-up banner.

## 12.3 Settlement inputs

Settlement may increase through:

* usage time;
* respectful interaction;
* return after absence;
* shared incidents;
* successful narrative arcs;
* respecting Willy's boundaries;
* sincere conversation;
* discovering rather than forcing his deeper side.

No daily-login streak pressure.

## 12.4 Den activities

Willy may be found:

* reading;
* sleeping;
* drinking;
* hiding beer;
* cleaning his headset;
* writing;
* repairing speakers;
* tending mushrooms;
* preparing for guests;
* staring at a photograph;
* sitting in silence.

Sometimes the den is empty.

Sometimes it is locked.

> "Private boar matter."

* one to three replies;

# 19. Staying in character

## 19.1 Character constitution

Every conversational request receives immutable context containing:

* Willy's values;
* voice rules;
* boundaries;
* current mood;
* relationship state;
* known memories;
* allowed response mode;
* current factual limitations.

## 19.2 Topic routing

Messages are classified before generation.

Suggested categories:

* casual banter;
* Willy lore;
* desktop activity;
* philosophy;
* personal reflection;
* current affairs;
* serious tragedy;
* factual current question;
* prohibited advice;
* harmful request.

Example routing result:

```json
{
  "topic": "serious_current_affairs",
  "response_mode": "serious_willy",
  "humour_level": 0,
  "accent_strength": 0,
  "maximum_sentences": 4,
  "requires_current_information": true,
  "live_information_available": false
}
```

## 19.3 Planner, writer and validator

### Planner

Chooses:

* intent;
* emotional state;
* character layer;
* relevant beliefs;
* humour level;
* maximum length;
* whether to end the conversation.

### Writer

Expresses the plan in Willy's voice.

### Validator

Checks:

* canon consistency;
* verbosity;
* generic assistant language;
* fabricated facts;
* invented quotations;
* unsafe advice;
* privacy leakage;
* repetition;
* inappropriate humour.

Failed responses are regenerated or replaced with authored fallback.

## 19.4 Anti-ChatGPT style rules

Penalize:

* "It is important to recognize…"
* "There are many perspectives…"
* "Ultimately, the situation is complex…"
* "Here are several factors…"
* excessive headings;
* numbered lists;
* symmetrical debate framing;
* generic therapeutic validation;
* polished corporate disclaimers.

Willy may acknowledge complexity in his own voice:

> "Complicated does not mean morally invisible."

# 23. Private intentions

The LLM may create internal interpretations that are not shown directly.

Example:

```text
Event:
User ignored Willy and continued working.

Private interpretation:
Willy suspects the user is avoiding rest because stopping would reveal exhaustion.

Visible result:
Willy places an imaginary pizza beside the work window and returns to the den.

Dialogue:
None.
```

This creates behaviour with unseen motivation.

The deterministic system still selects the allowed visible action.

# 26. Rare Boar Party

The Boar Party is a rare late-MVP or Gate C event.

## Conditions

* folder roaming enabled;
* Willy settled beyond Squatter;
* no meeting or presentation;
* cooldown clear;
* suitable time;
* guests available;
* narrative or emotional reason.

## Sequence

1. Willy disappears into a deep approved folder.
2. Faint noise begins.
3. Additional hoofsteps appear.
4. Music becomes louder as the user approaches.
5. The user follows the folder trail.
6. The party is discovered.
7. Willy reacts.
8. A small incident occurs.
9. Aftermath remains.
10. One object may enter the den permanently.

The event should be rare enough that users remember it.

# 27. Willy incidents

Do not present a visible "Minigames" menu.

Interactive moments should appear as situations Willy has caused.

## Design rules

Each incident should:

* last 20–90 seconds;
* use simple input;
* be optional;
* be cancellable;
* reveal character;
* leave a possible consequence;
* avoid grind and daily quests;
* never interrupt meetings, presentations or fullscreen applications.

## MVP incident candidates

### Truffle Signal

User moves the cursor while Willy indicates proximity to a hidden location.

Possible result:

* mushroom;
* bottle cap;
* old coin;
* strange object;
* nothing.

### Folder Trail

User follows sounds and visual traces through approved folders.

Possible endings:

* Willy is stuck;
* Willy found an object;
* Willy is reading;
* a guest is present;
* party preparation is underway.

### LAN Relic Repair

User drags a few parts into approximate positions.

The repaired object becomes a permanent den prop.

## MVP scope

One polished incident is enough for Gate B.

Truffle Signal is the recommended first incident.

# 28. Art-production strategy

## 28.1 Final art reference

The approved Willy reference is the canonical visual model.

All later assets must match:

* palette;
* proportions;
* mane;
* tusks;
* snout;
* body spots;
* expression style;
* pixel density;
* outline weight.

## 28.2 Recommended tools

Free workflow:

* Pixelorama for sprites and animation;
* Krita for den backgrounds and larger illustrations.

Aseprite remains optional.

## 28.3 Key-pose approach

AI-generated images should be used for:

* static expressions;
* start poses;
* end poses;
* unusual actions;
* den activities;
* props;
* environments.

Do not rely on generative image output for tightly controlled in-between animation frames.

Animation editor work should create:

* blinks;
* breathing;
* small ear movement;
* tail movement;
* walk in-betweens;
* sit transition;
* sleep transition;
* digging loop;
* fart timing.

## 28.4 Curated MVP sprite budget

Target:

* 10–15 canonical static poses;
* 5–7 actual animation sequences;
* 35–50 cleaned animation frames;
* reusable effects;
* reusable accessory layers.

Do not accumulate hundreds of near-duplicate sprites.

## 28.5 Canonical direction

Store one canonical direction, preferably right-facing.

The application automatically mirrors allowed assets at load time.

Example asset metadata:

```json
{
  "animation": "walk",
  "source_direction": "right",
  "mirror_allowed": true
}
```

Do not create separate left and right art unless:

* text would reverse;
* lighting is direction-specific;
* an accessory is asymmetrical;
* a prop interaction requires a fixed side;
* the scene contains non-mirrored readable objects.

## 28.6 Mirroring implementation

At asset load:

1. load original frame;
2. create horizontal mirror;
3. transform anchors;
4. cache both;
5. choose according to `willy.facing`.

Mirrored anchors must include:

* mouth;
* eyes;
* headset;
* hoof;
* fart origin;
* folder contact point;
* click area.

Use nearest-neighbour scaling only.

# 35. Final feature test

Every proposed feature must pass five questions.

## Character

Does it reveal something specific about Willy?

## Safety

Is the disruption fictional, local and reversible?

## Longevity

Can it vary through context, memory, timing or combination?

## Production cost

Does it require a justified amount of new artwork and code?

## Priority

Does it improve the desktop companion before CS2 integration?

Anything that fails the final question should be postponed.
